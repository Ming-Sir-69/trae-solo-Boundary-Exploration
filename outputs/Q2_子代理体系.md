# Q2：Trae SOLO 子代理体系详解

## 一、概述

Trae SOLO 的主 Agent（Main Agent）下共有 **3 类子代理（Subagent）**，通过 `Task` 工具调度。子代理在独立上下文中运行，完成后向主 Agent 回传总结结果。

---

## 二、三类子代理详解

### 2.1 Explore Subagent（探索检索型子代理）

| 属性 | 说明 |
|------|------|
| **类型标识** | `subagent_type: "Explore"` |
| **核心能力** | 快速探索与研究任务 |
| **可用工具** | LS、Read、Glob、Grep、WebSearch、WebFetch、RunCommand（只读）、浏览器工具 |
| **写入权限** | ❌ 只读（不能创建/修改/删除文件） |
| **适用场景** | 扫描代码库、定位文件与调用链、检索外部资料、理解项目架构、多源信息综合 |

**典型使用场景**：
- 场景 A：代码库探索 + 外部资料并行检索
- 场景 B：多模块同时摸底（各模块目录/功能独立）
- 场景 D：多报错/多现象并行定位根因

### 2.2 Plan Subagent（方案规划型子代理）

| 属性 | 说明 |
|------|------|
| **类型标识** | `subagent_type: "Plan"` |
| **核心能力** | 实现策略设计与步骤规划 |
| **可用工具** | LS、Read、Glob、Grep、WebSearch、WebFetch、RunCommand（只读）、浏览器工具 |
| **写入权限** | ❌ 只读（不能创建/修改/删除文件） |
| **适用场景** | 设计实施策略、创建步骤计划、评估技术方案、识别架构权衡 |

**典型使用场景**：
- 场景 C：多方案并行评估（A/B/C 路线，每条输出优缺点、依赖、风险）
- 场景 A/B 的辅助：摸底 + 方案设计

### 2.3 Coding Subagent（编码执行型子代理）

| 属性 | 说明 |
|------|------|
| **类型标识** | `subagent_type: "general_purpose_task"` |
| **核心能力** | 通用编码任务执行 |
| **可用工具** | **全部工具**（包括 Write、Edit、DeleteFile、RunCommand 等） |
| **写入权限** | ✅ 可写入（创建/修改/删除文件） |
| **适用场景** | 实现代码、运行脚本、执行自动化任务、多步骤编码实现 |

**典型使用场景**：
- 场景 E：多模块互不冲突的代码块并行实现（**前提：接口已冻结，改动不重叠**）

## 三、并行调度规则

### 3.1 并行触发的 5 条硬条件（必须同时满足）

| 序号 | 条件 | 说明 |
|------|------|------|
| 1 | **可拆分** | 任务可拆成 ≥2 个子任务，每个子任务有明确产出 |
| 2 | **弱依赖/无依赖** | 子任务推进不需要彼此的中间结果（或只需粗粒度信息） |
| 3 | **收益足够** | 预计节省时间 ≥30% |
| 4 | **一次性指令可闭环** | 子代理能在"单轮输入→最终回传总结"模式下完成 |
| 5 | **写入不冲突** | 并行阶段默认只做只读/规划；若必须写入，保证不同子代理不改同一文件 |

### 3.2 禁止并行的 5 种情况（命中任一则串行）

| 序号 | 情况 | 说明 |
|------|------|------|
| 1 | **强依赖链** | 必须先完成 A 才能开始 B（如先修编译再跑测试） |
| 2 | **需求未澄清** | 目标/边界不清晰，分出去只会产生分叉与返工 |
| 3 | **高冲突写入风险** | 多个子代理会改同一文件/同一核心模块 |
| 4 | **需要频繁交互确认** | 子代理因缺关键参数而只能做假设 |
| 5 | **SSOT/历史决策必须先对齐** | 涉及关键架构决策或既有坑记录时，先主线程读档对齐 |

### 3.3 并行上限

| 场景 | 上限 |
|------|------|
| 默认并行 | **2 个**子代理 |
| 多模块/多方案评估 | 最多 **3 个** |
| 超过 3 个 | 必须有明确拆分边界，否则收敛成本反噬效率 |

### 3.4 默认安全并行组合

| 组合 | 安全级别 | 说明 |
|------|---------|------|
| Explore + Explore | ⭐⭐⭐ 最安全 | 纯只读，零冲突风险 |
| Plan + Plan | ⭐⭐⭐ 最安全 | 纯只读，零冲突风险 |
| Explore + Plan | ⭐⭐⭐ 最安全 | 纯只读，零冲突风险 |
| Explore + Plan + Explore | ⭐⭐ 安全 | 三路只读，收敛成本稍高 |
| Coding + Coding | ⚠️ 有条件 | **仅在接口冻结 + 改动不重叠时允许** |

## 四、收敛规则（Single Source of Truth）

1. **主线程必须"去重+冲突消解"**，输出单一实施路线
2. **证据优先**：子代理结论冲突时，以代码位置/日志/官方文档原文为依据
3. **写入唯一性**：最终写入只允许主线程写入，或单一 Coding Subagent 负责写入（其他只读给建议）

---

## 五、自动触发决策树

```
开始
  │
  ├─ 子任务数 N < 2？ → 串行
  │
  ├─ 存在强依赖？ → 串行
  │
  ├─ 存在写入冲突？ → 改为"只读并行 + 单点写入"或串行
  │
  ├─ 节省时间 < 30%？ → 串行
  │
  └─ 以上均否 → 自动并行（默认 2 个子代理起）
```

## 六、本次任务中的实际并行验证

在本次"Trae SOLO 能力测试"项目中，实际演示了以下并行调度：

### 6.1 信息收集阶段（3 路并行）
- **Explore Subagent A**：扫描系统配置和模式信息（/mnt/appuserdata/、/sessions/）
- **Explore Subagent B**：搜索 Trae SOLO 官方网页信息（trae.cn、docs.trae.ai）
- **Explore Subagent C**：探索文件预览能力和完整 Skill 清单

### 6.2 执行阶段（2 路并行 × 2 批）
- **第一批**：
  - **Coding Subagent A**（Q1）：撰写模式与场景分析文档
  - **Coding Subagent B**（Q4）：执行文件预览测试（创建/下载 22 个测试文件）
- **第二批**（主线程）：
  - Q2 子代理体系（本文档）
  - Q3 工具清单

### 6.3 验证结论
- ✅ Explore + Explore 并行：稳定可靠，零冲突
- ✅ Coding + Coding 并行：在不同输出目录下成功并行（Q1 和 Q4 写入不同文件）
- ✅ 并行效率：信息收集阶段 3 路并行大幅缩短调研时间

## 七、子代理 vs 用户模式的区别

| 维度 | 子代理（Subagent） | 用户模式（Mode） |
|------|-------------------|-----------------|
| 谁触发 | 主 Agent 自动调度 | 用户在 UI 中手动切换 |
| 用户感知 | 用户看到主 Agent 在执行任务 | 用户主动选择进入某个模式 |
| 数量 | 3 类（Explore/Plan/Coding） | 2 个顶层（Code/MTC）+ 2 个功能级（Plan/Spec） |
| 上下文 | 独立上下文，完成后回传摘要 | 共享用户会话上下文 |
