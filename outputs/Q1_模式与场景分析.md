## 一、模式体系概述

### 1.1 两层嵌套结构

Trae SOLO 桌面版的模式体系采用 **"顶层模式 + 内嵌功能"** 的两层嵌套结构：

```
Trae SOLO 桌面版
├── Code 模式（面向开发者）
│   ├── Plan 功能（先规划后执行）
│   └── Spec 功能（需求规格化，Code 模式独有）
│
└── MTC 模式（More Than Code，面向非开发者）
    └── Plan 功能（先规划后执行）
```

**第一层：顶层模式（用户可选）**

| 模式 | 全称 | 目标用户 | 核心定位 |
|------|------|----------|----------|
| **Code** | Code Mode | 开发者、技术岗 | 编码执行、脚本运行、自动化工程 |
| **MTC** | More Than Code | 产品经理、分析师、运营、学生等 | 文档/表格/报告/PPT 等全产研场景 |

**第二层：内嵌功能（模式内可选）**

| 功能 | 所属模式 | 说明 |
|------|----------|------|
| **Plan** | Code + MTC | 先拆解任务步骤，再逐步执行；适合复杂多步骤任务 |
| **Spec** | 仅 Code | 将自然语言需求转化为结构化规格文档，作为编码的契约基线 |

### 1.2 模式选择逻辑

```
用户打开 Trae SOLO
    │
    ├─ 任务涉及写代码/跑脚本/调试程序？
    │     └─ 是 → Code 模式
    │
    ├─ 任务涉及文档/表格/PPT/报告/数据分析？
    │     └─ 是 → MTC 模式
    │
    └─ 不确定？
          └─ MTC 模式（覆盖面更广，可兜底）
```

## 二、Code 模式详解

### 2.1 模式定位

Code 模式是 Trae SOLO 面向 **开发者和技术岗位用户** 的核心模式。其能力核心在于：

- **编码执行**：支持 Python、JavaScript/TypeScript、Shell 等主流语言的编写与运行
- **脚本自动化**：批量文件处理、数据清洗、日志分析等自动化任务
- **工程化能力**：项目初始化、依赖安装、测试运行、Git 操作
- **Spec 需求规格化**：将模糊需求转化为可执行的技术规格（Code 模式独有）

### 2.2 场景 C1：React + Tailwind CSS 搭建电商商品列表页

| 维度 | 内容 |
|------|------|
| **行业** | 互联网 AI 软件行业 |
| **岗位** | 初级前端开发实习生 |
| **人员类别** | 开发者 |
| **职级** | 实习级（入门） |
| **适合模式** | Code 模式 |

#### 背景

某电商创业公司刚拿到天使轮融资，产品团队急需一个商品展示页面的 MVP。团队只有 3 名前端开发，其中 1 名是刚入职两周的实习生小李。技术栈已确定为 React + Tailwind CSS，但资深同事都在忙核心交易模块，没有时间手把手带小李。小李需要在半天内独立完成商品列表页的搭建，包括商品卡片展示、分类筛选、响应式布局等基础功能。

#### 任务描述

使用 React + Tailwind CSS 搭建一个电商商品列表页，要求：
1. 商品卡片展示（图片、名称、价格、标签）
2. 顶部搜索栏 + 分类筛选标签
3. 响应式布局（手机 1 列、平板 2 列、桌面 4 列）
4. 使用 Mock 数据（至少 12 个商品）

#### 操作步骤

1. **启动 Code 模式**，输入需求描述
2. **Plan 功能**（可选）：让 Trae SOLO 先规划项目结构、组件拆分方案
3. **Spec 功能**（可选）：将需求规格化，明确组件 API、数据结构、样式规范
4. Trae SOLO 自动执行：
   - 初始化 React 项目（Vite 脚手架）
   - 安装 Tailwind CSS 依赖
   - 创建组件文件（ProductCard、SearchBar、CategoryFilter、ProductGrid）
   - 编写 Mock 数据
   - 实现响应式布局
   - 运行开发服务器预览

#### 预期输出

- 一个可运行的 React 项目目录
- 商品列表页组件代码（`.tsx` 文件）
- Mock 数据文件（`products.ts`）
- 开发服务器启动后可在浏览器预览完整页面
- 响应式布局在不同屏幕尺寸下正常展示

### 2.3 场景 C2：Python 脚本读取设备日志 CSV 识别异常温度报警

| 维度 | 内容 |
|------|------|
| **行业** | 生产制造业 |
| **岗位** | 车间设备维修工 |
| **人员类别** | 纯体力劳动者 |
| **职级** | 一线操作工（无编程基础） |
| **适合模式** | Code 模式 |

#### 背景

某汽车零部件工厂的冲压车间有 20 台数控机床，每台设备每 5 分钟生成一条温度日志记录，保存为 CSV 文件。车间维修工老王负责设备巡检，每天需要手动翻看数百条日志来判断是否有设备温度异常。老王只有初中学历，完全不懂编程，但他知道"主轴温度超过 85 度就要报修"。目前他每天花 2 小时翻日志，经常漏看。

#### 任务描述

用自然语言告诉 Trae SOLO："帮我写一个 Python 脚本，读取设备日志 CSV 文件，找出温度超过 85 度的记录，按设备编号分组汇总，输出一个简单的报警清单。"

#### 操作步骤

1. **启动 Code 模式**，用大白话描述需求
2. Trae SOLO 自动执行：
   - 分析 CSV 文件结构（列名、数据格式）
   - 编写 Python 脚本（使用 pandas 读取 CSV、筛选异常记录、分组汇总）
   - 运行脚本并输出报警清单
3. 老王拿到结果后，直接去对应设备处检查

#### 预期输出

- 一份 Python 脚本（`.py` 文件）
- 脚本运行后的报警清单，格式示例：

```

  - 08:20  主轴温度: 89.1°C
  - 09:10  主轴温度: 86.5°C
  - 10:05  主轴温度: 91.2°C  ← 最高
  - 13:40  主轴温度: 85.8°C

[设备 NC-017] 异常 2 次
  - 11:30  主轴温度: 86.0°C
  - 14:15  主轴温度: 88.7°C

共 2 台设备异常，请尽快安排检修。

- 后续老王只需双击运行脚本，无需任何编程知识

### 2.4 场景 C3：批量重命名脚本归档快递面单图片

| **岗位** | 快递站点分拣员 |
| **人员类别** | 纯体力劳动者 |

#### 任务描述

用自然语言告诉 Trae SOLO："我有一堆快递面单照片，文件名都是 IMG 开头的乱码。帮我写个脚本，从照片里识别出快递单号，然后把文件重命名为'单号_日期.jpg'，再按日期放到不同文件夹里。"

   - 批量处理全部照片
3. 小张拿到整理好的文件夹，直接用于存档

#### 预期输出

- 一份 Python 脚本（`.py` 文件）
- 按日期归档的文件夹结构：

```

│   └── JD0029384756_2026-04-17.jpg
├── 2026-04-18/
│   ├── YT9283746502_2026-04-18.jpg
│   ├── SF1093827457_2026-04-18.jpg
│   └── ...

- 处理日志（记录哪些文件识别成功、哪些失败需人工复核）
- 后续小张只需把新照片丢进指定文件夹，运行脚本即可

## 三、MTC 模式详解

### 3.1 模式定位

MTC（More Than Code）模式是 Trae SOLO 面向 **非技术岗位用户** 的核心模式。其能力核心在于：

- **文档生成与编辑**：Word 文档、PDF 处理、Markdown 文档
- **表格处理**：Excel/CSV 数据整理、公式计算、数据透视、图表生成
- **演示文稿**：PPT 制作、幻灯片设计、数据可视化嵌入
- **数据分析与报告**：数据清洗、统计分析、趋势洞察、汇报材料
- **多格式转换**：图片转文字、PDF 提取、跨格式数据迁移

MTC 模式的核心价值是 **让不会写代码的人也能完成数据驱动的办公任务**。

### 3.2 场景 M1：手写销售记录图片整理成 Excel + 趋势图

| 维度 | 内容 |
|------|------|
| **行业** | 服务业（餐饮零售） |
| **岗位** | 咖啡店兼职大学生 |
| **人员类别** | 学生办公族 |
| **职级** | 基层兼职（学生） |
| **适合模式** | MTC 模式 |

#### 背景

大学城附近的一家独立咖啡店，老板习惯用纸笔记录每日销售数据（写在便签纸上：日期、品名、数量、金额）。兼职大学生小赵每周日需要把这些手写记录整理成电子表格，交给老板看本周经营情况。小赵每周要花 3 小时手动录入数据，经常因为字迹潦草看错数字。老板最近想看"哪款咖啡卖得最好"的趋势分析，但小赵不会用 Excel 做图表。

#### 任务描述

把手写销售记录的照片交给 Trae SOLO，要求：
1. 识别照片中的手写内容，提取日期、品名、数量、金额
2. 整理成规范的 Excel 表格
3. 生成销售趋势图（按品名汇总，折线图或柱状图）

#### 操作步骤

1. **启动 MTC 模式**，上传手写销售记录照片
2. 用自然语言描述需求："帮我把这些手写记录整理成 Excel，再画个趋势图"
3. Trae SOLO 自动执行：
   - OCR 识别手写内容
   - 结构化数据提取（日期、品名、数量、金额）
   - 生成 Excel 表格（含汇总行、格式美化）
   - 生成销售趋势图表
   - 打包输出

#### 预期输出

- 一份 Excel 文件（`.xlsx`），包含：
  - **原始数据 Sheet**：逐条录入的销售记录
  - **汇总 Sheet**：按品名汇总的周销售统计
  - **趋势图**：各品名销售量/销售额的柱状图或折线图

| 日期 | 品名 | 数量 | 金额(元) |
|------|------|------|----------|
| 4/14 | 美式咖啡 | 23 | 460 |
| 4/14 | 拿铁 | 31 | 930 |
| 4/15 | 美式咖啡 | 19 | 380 |
| ... | ... | ... | ... |

- 小赵每周只需拍照上传，5 分钟搞定原来 3 小时的活

### 3.3 场景 M2：多份 PDF 入库单提取到统一 Excel 模板

| 维度 | 内容 |
|------|------|
| **行业** | 生产制造业 |
| **岗位** | 仓库管理员 |
| **人员类别** | 纯体力劳动者 |
| **职级** | 一线操作工（无编程基础） |
| **适合模式** | MTC 模式 |

#### 背景

某电子元器件工厂的仓库每天收到来自不同供应商的入库单，格式各异——有的供应商发 PDF，有的发 Word，有的发拍照图片。仓库管理员老陈需要把这些入库单上的信息（物料编号、名称、规格、数量、批次号）手动录入到公司的统一 Excel 入库台账中。老陈每天处理 30-50 份入库单，手动录入容易出错，一旦录错物料编号会导致后续生产领料混乱。

#### 任务描述

把多份不同格式的入库单文件交给 Trae SOLO，要求：
1. 从 PDF/Word/图片中提取关键信息
2. 按照公司统一 Excel 模板格式输出
3. 自动校验物料编号格式是否正确

1. **启动 MTC 模式**，上传所有入库单文件 + 公司 Excel 模板
2. 用自然语言描述需求："帮我把这些入库单的信息提取出来，填到这个模板里"

   - 解析 PDF/Word/图片内容
   - 识别并提取关键字段（物料编号、名称、规格、数量、批次号）
   - 按模板格式填入 Excel
   - 校验物料编号格式
   - 标记无法识别或格式异常的条目

#### 预期输出

- 一份填写完成的 Excel 入库台账（`.xlsx`），格式与公司模板一致：

| 序号 | 入库日期 | 供应商 | 物料编号 | 物料名称 | 规格 | 数量 | 单位 | 批次号 | 备注 |

| 1 | 2026-04-18 | 华芯电子 | R0402-10K | 贴片电阻 | 0402/10KΩ | 50000 | pcs | B20260418-01 | |
| 2 | 2026-04-18 | 华芯电子 | C0603-100N | 贴片电容 | 0603/100nF | 30000 | pcs | B20260418-02 | |
| 3 | 2026-04-18 | 信达光电 | LED-0603-R | 红色LED | 0603 | 10000 | pcs | B20260418-03 | ⚠️ 批次号格式待确认 |

- 异常标记：对识别不确定的字段高亮标注，提醒老陈人工复核
- 老陈每天只需上传文件，原来 4 小时的录入工作缩短到 15 分钟

### 3.4 场景 M3：用户反馈 Excel 生成月度运营汇报 PPT

| 维度 | 内容 |
|------|------|
| **行业** | 互联网 AI 软件行业 |
| **岗位** | 运营专员 |
| **人员类别** | 学生办公族 |
| **职级** | 基层专员（入职 1 年） |
| **适合模式** | MTC 模式 |

#### 背景

某 AI 工具类 SaaS 产品的运营团队每月需要向管理层提交运营汇报。运营专员小林负责收集用户反馈（来自客服系统导出的 Excel，包含反馈时间、用户等级、反馈类型、反馈内容、处理状态等字段），并制作月度汇报 PPT。小林每月要花 2 天做 PPT：先用 Excel 做数据透视和图表，再手动复制到 PPT 里排版。她不擅长数据可视化，做出来的图表总是被领导嫌弃"不够专业"。

#### 任务描述

把用户反馈 Excel 文件交给 Trae SOLO，要求：
1. 分析用户反馈数据，提取关键运营指标
2. 生成数据可视化图表
3. 制作一份专业的月度运营汇报 PPT（含数据图表、趋势分析、改进建议）

#### 操作步骤

1. **启动 MTC 模式**，上传用户反馈 Excel 文件
2. 用自然语言描述需求："帮我分析这份用户反馈数据，做一份月度运营汇报 PPT"
3. Trae SOLO 自动执行：
   - 读取并分析 Excel 数据
   - 计算关键指标（反馈总量、各类型占比、处理率、用户等级分布、月度趋势等）
   - 生成数据可视化图表（饼图、柱状图、折线图等）
   - 制作 PPT 幻灯片（封面、数据概览、分类分析、趋势洞察、改进建议）
   - 输出完整 PPT 文件

#### 预期输出

- 一份月度运营汇报 PPT（`.pptx`），包含以下幻灯片：

| 页码 | 内容 |
|------|------|
| 第 1 页 | 封面：2026 年 4 月用户反馈运营月报 |
| 第 2 页 | 核心指标概览：反馈总量、处理率、平均响应时间、满意度 |
| 第 3 页 | 反馈类型分布饼图（Bug 反馈/功能建议/使用咨询/投诉） |
| 第 4 页 | 月度反馈趋势折线图（近 6 个月对比） |
| 第 5 页 | 用户等级分布柱状图（免费用户/基础版/专业版/企业版） |
| 第 6 页 | Top 10 高频反馈内容词云或列表 |
| 第 7 页 | 未处理反馈清单及责任人 |
| 第 8 页 | 改进建议与下月计划 |

- 图表配色专业、排版美观，达到可直接向管理层汇报的水准
- 小林每月只需上传数据文件，原来 2 天的工作缩短到 30 分钟

| **行业** | 服务业、生产制造业、互联网 AI 软件行业 | 服务业（C3、M1）、生产制造业（C2、M2）、互联网 AI 软件（C1、M3） | 全覆盖 |
| **人员类别** | 开发者、学生办公族、纯体力劳动者 | 开发者（C1）、学生办公族（M1、M3）、纯体力劳动者（C2、C3、M2） | 全覆盖 |
| **模式** | Code 模式、MTC 模式 | Code（C1、C2、C3）、MTC（M1、M2、M3） | 全覆盖 |
| **职级跨度** | 实习级 ~ 一线操作工 ~ 基层专员 | 实习生、一线维修工、一线分拣员、兼职学生、仓库管理员、运营专员 | 跨度充分 |

### 4.3 核心差异对比

| 对比维度 | Code 模式 | MTC 模式 |
|----------|-----------|----------|
| **目标用户** | 有/无编程基础均可（偏技术场景） | 无编程基础（偏办公场景） |
| **输入方式** | 自然语言描述 + 代码文件 | 自然语言描述 + 文档/表格/图片 |
| **核心产出** | 可执行代码、脚本、自动化工具 | Excel、PPT、Word、PDF、数据报告 |
| **技术门槛** | 用户无需写代码，但需理解技术概念 | 用户完全无需技术背景 |
| **典型场景** | 网页开发、数据处理脚本、文件自动化 | 数据整理、报告制作、格式转换 |
| **独有功能** | Spec（需求规格化） | — |
| **共有功能** | Plan（先规划后执行） | Plan（先规划后执行） |

## 五、Plan 和 Spec 功能说明

### 5.1 Plan 功能（Code + MTC 共有）

**功能定义**：Plan 是嵌套于两种模式中的任务规划功能。启用后，Trae SOLO 会先对用户需求进行拆解和规划，生成一个分步骤的执行计划，经用户确认后再逐步执行。

**适用场景**：
- 任务步骤较多（超过 3 步）
- 任务逻辑复杂，需要先理清思路
- 用户希望对执行过程有掌控感
- 多文件/多模块的协同任务

**工作流程**：

```
用户输入需求
    │
    ▼
Plan 模式启动
    │
    ├─ 1. 需求理解：解析用户意图，确认任务边界
    ├─ 2. 任务拆解：将复杂任务拆分为可执行的子步骤
    ├─ 3. 依赖分析：确定步骤间的先后顺序
    ├─ 4. 风险识别：标注可能的卡点和备选方案
    │
    ▼
输出执行计划（供用户审阅）
    │
    ├─ 用户确认 → 逐步执行
    └─ 用户修改 → 调整计划 → 再次确认 → 执行
```

**与不使用 Plan 的区别**：

| 维度 | 不使用 Plan | 使用 Plan |
|------|-------------|-----------|
| 执行方式 | 直接执行，一步到位 | 先规划，再分步执行 |
| 可控性 | 较低（AI 自主决策） | 较高（用户确认每一步） |
| 适用任务 | 简单、明确的单步任务 | 复杂、多步骤的复合任务 |
| 中间反馈 | 无（直接给最终结果） | 有（每步完成后可检查） |

### 5.2 Spec 功能（仅 Code 模式）

**功能定义**：Spec 是 Code 模式独有的需求规格化功能。它将用户的自然语言需求转化为结构化的技术规格文档（Specification），作为后续编码执行的契约基线。

**适用场景**：
- 需求描述模糊，需要先明确技术细节
- 团队协作场景，需要统一理解需求
- 复杂工程任务，需要先定义接口/数据结构/验收标准
- 需求可能变更，需要留存基线版本

**Spec 文档典型结构**：

```markdown
# 需求规格文档（Spec）

## 1. 项目概述
- 项目名称、背景、目标

## 2. 功能需求
- 功能点列表、优先级、验收标准

## 3. 技术方案
- 技术栈选型、架构设计、目录结构

## 4. 接口定义
- API 接口、数据结构（DTO/Schema）、组件 Props

## 5. 约束与假设
- 性能要求、兼容性要求、已知限制

## 6. 里程碑
- 阶段划分、交付物清单
```

**Spec 与 Plan 的协作关系**：

```
用户需求
    │
    ▼
Spec（需求规格化）──→ 明确"做什么"
    │
    ▼
Plan（任务规划）──→ 明确"怎么做、分几步"
    │
    ▼
Code 执行 ──→ 产出最终代码/文件
```

- **Spec** 回答的是 "What"（做什么、做到什么标准）
- **Plan** 回答的是 "How"（怎么做、分几步、先做哪步）
- 两者可以串联使用，也可以单独使用

### 5.3 功能选择建议

| 场景特征 | 推荐组合 | 原因 |
|----------|----------|------|
| 简单明确的单步任务 | 直接执行（不启用 Plan/Spec） | 过度规划浪费时间 |
| 多步骤但需求清晰 | Plan | 需要分步执行和中间确认 |
| 需求模糊的编码任务 | Spec + Plan | 先明确需求，再规划执行 |
| 团队协作的工程任务 | Spec + Plan | 需要留存规格文档供团队对齐 |
| 非技术用户的办公任务 | Plan（MTC 模式） | 用户需要了解 AI 的处理思路 |
